On-Demand Self-Service

POST /detect-face triggers a cloud worker on Render to run DeepFace. No admin needs to start anything. POST /detect-voice spins up Librosa MFCC processing instantly. Resources scale with load.

Broad Network Access
Services available over network, accessed via standard mechanisms from any client platform.
Vercel hosts the React frontend on a global CDN. Render exposes the FastAPI backend on a public HTTPS URL. A user in Odisha and a user in Berlin access the same system via standard HTTP from any browser.

🔄
Resource Pooling
Provider resources pooled to serve multiple consumers. Location independent.
MongoDB Atlas serves all users from a shared managed cluster (collections + GridFS). Render runs the FastAPI process on shared cloud compute. Multiple users' requests are handled by the same worker pool transparently.
app.add_middleware(CORSMiddleware,
  allow_origins=["*"])
# Any user, any origin
04
📈
Rapid Elasticity / Scalability
Capabilities scale out/in rapidly, appearing unlimited to the consumer.
FastAPI is stateless — Render can add worker instances with zero code change. Atlas scales with your cluster tier; GridFS stores large binaries in the same database. 100 simultaneous emotion detection requests are handled as easily as 1.
# No session, no in-memory state

05
📊
Measured Service
Cloud systems automatically control and optimise resources. Usage monitored and reported.
Every API call increments MongoDB counters via $inc on system_stats (total_requests, face_requests, voice_requests). Events go to the logs collection. Admin panel reads live stats from Atlas.
db.system_stats.update_one(
  {"_id": "usage"},
  {"$inc": {"total_requests": 1, "face_requests": 1}},
  upsert=True)
06
💾
Cloud Storage (Zero Local I/O)
Data stored in managed cloud infrastructure — no dependency on local disk.
File bytes arrive at FastAPI, stay in RAM, then GridFS.put() stores them on Atlas. A public API URL (/files/{id}) is saved on the emotion document. No local filesystem. Restart the server — no data loss. Data lives in the managed database.

🔀 Weighted Emotion Fusion
utils/emotion_combiner.py
# Face is more reliable — higher weight
FACE_WEIGHT  = 0.60
VOICE_WEIGHT = 0.40

for emotion in EMOTIONS:
    combined[emotion] = (
        FACE_WEIGHT  * face_scores[emotion] +
        VOICE_WEIGHT * voice_scores[emotion]
    )

dominant = max(combined, key=combined.get)
# e.g. face=happy(0.8), voice=happy(0.6) → combined=happy(0.72)
# e.g. face=happy(0.8), voice=sad(0.7)  → combined=happy(0.52)

Cloud Deployment
Deployment Steps
Deploy all three tiers to cloud in under 30 minutes
🍃
MongoDB Atlas
Cloud DB + GridFS · Managed cluster
  1. Create a free (M0) cluster at mongodb.com/cloud/atlas
  2. Database Access → create user + password
  3. Network Access → allow your IP (or 0.0.0.0/0 for dev only)
  4. Connect → Drivers → copy SRV connection string
  5. Set MONGODB_URI and MONGODB_DB_NAME in backend env
  6. Set API_PUBLIC_URL to your API’s public HTTPS URL (same origin as media links)
⚡
Backend → Render
FastAPI Cloud API · On-Demand Compute
  1. Push backend/ folder to GitHub
  2. render.com → New → Web Service
  3. Connect GitHub repo, select backend folder
  4. Build: pip install -r requirements.txt
  5. Start: uvicorn main:app --host 0.0.0.0 --port $PORT
  6. Add env vars: MONGODB_URI, MONGODB_DB_NAME, API_PUBLIC_URL
  7. Deploy → copy public URL
🌐
Frontend → Vercel
React CDN · Global Edge Network
  1. Push frontend/ folder to GitHub
  2. vercel.com → New Project → Import repo
  3. Framework preset: Vite
  4. Set VITE_API_URL = your Render backend URL (match API_PUBLIC_URL host for images/audio)
  5. Deploy → get public Vercel URL
  6. Share URL — accessible globally 🌍

Exam Preparation
Viva Q&A Guide
Common viva questions with answers specific to this project — not generic theory
Q1Why is this system called "cloud-based"? What makes it different from a local system? ▾
In a local system, the ML model, database, and files would all run on one machine. In our system:

1. The FastAPI backend with DeepFace and Librosa runs on Render's cloud servers — not our laptop.
2. Image and audio binaries are stored in MongoDB GridFS on Atlas (no local disk on the API server).
3. Metadata (emotion, timestamp, file URL) is in MongoDB collections (emotions, logs, system_stats).
4. The React frontend is hosted on Vercel's global CDN.

No data touches any local machine. The system works from any device anywhere in the world via the internet.
Q2How does your system achieve scalability? ▾
Our FastAPI backend is stateless — it stores no data in server memory between requests. Every request carries all necessary information. This is the key to scalability.

Because there is no shared state, Render can run multiple instances of our FastAPI server behind a load balancer. 100 simultaneous users can each get their own worker process.

MongoDB Atlas is a managed cluster — scale by choosing a larger tier; the same PyMongo API works for one user or many.

GridFS stores large files inside the database; media is served via GET /files/{id} on the public API URL.
Q3Explain on-demand computing in your project with a specific example. ▾
When a user clicks "Capture & Detect" in the browser:

1. The image is sent via POST /detect-face to the Render cloud server.
2. The server runs DeepFace.analyze() on that image — this is the on-demand compute.
3. Result is returned and the server is idle again.

There is no continuous processing. Compute happens only when a request arrives. If no user sends a request, no CPU is used. This matches the NIST definition of on-demand self-service. The cloud platform provisions compute resources dynamically.
Q4How is cloud storage used? Does the server save files locally? ▾
No, the server never writes to local disk.

When an image arrives: file_bytes = await file.read() — this stays in RAM.
Then: GridFS.put(file_bytes, content_type=...) — binary goes to Atlas; file_url points to /files/{ObjectId} on the API.
The emotion document stores that URL in MongoDB.

If the Render server restarts, no data is lost because data lives in Atlas. This is the cloud storage principle — persistence is in a managed service, not on the compute instance disk.
Q5What is the background task in your system and why is it a cloud feature? ▾
We use APScheduler inside the FastAPI process to run a daily cleanup job at 02:00 UTC.

This job: reads auto_delete_enabled from system_stats in MongoDB. If enabled, soft-deletes old emotions and removes matching GridFS files.

This is a cloud feature because: (1) it runs on the cloud server 24/7, not on any user's machine. (2) Admin can toggle it remotely via POST /admin/toggle-auto-delete — no SSH needed. (3) It operates on Atlas (collections + GridFS) only.
Q6How does your system reduce hardware dependency? ▾
The user's device only needs a browser, camera, and microphone.

All heavy computation runs on the cloud:
• DeepFace (deep learning face analysis) → Render server
• Librosa MFCC extraction (audio DSP) → Render server
• Database queries → MongoDB Atlas

A budget Android phone with Chrome can use this system. No Python, no GPU, no installation required on the client side. This is the "reduced hardware dependency" principle of cloud computing.
Q7What are MFCC features? How do they help in voice emotion detection? ▾
MFCC = Mel-Frequency Cepstral Coefficients. They represent the short-term power spectrum of audio on a non-linear mel scale that approximates human hearing.

In our system, we extract 40 MFCC coefficients from each audio clip using Librosa, then compute their mean and standard deviation across time frames. We also extract:
• Zero Crossing Rate (ZCR) — how often signal crosses zero (indicates voice activity)
• Spectral Centroid — brightness/pitch proxy
• Spectral Rolloff — frequency distribution
• Chroma Features — tonal content
• Tempo — speech rate (angry = faster, sad = slower)

These features together distinguish emotional speech patterns.
Q8How does the weighted emotion fusion work? ▾
The system runs face detection and voice detection independently. Each returns an emotion with a confidence score.

Then POST /combine-emotions applies weighted averaging:
combined[emotion] = 0.60 × face_score[emotion] + 0.40 × voice_score[emotion]

Face gets 60% weight because facial expressions are more consistently labelled. Voice gets 40%.

Example: Face = happy (80% confidence), Voice = happy (60% confidence)
Combined happy = 0.6×0.8 + 0.4×0.6 = 0.48 + 0.24 = 0.72 (72%)

The system also detects agreement (both say same emotion) vs mixed signals (face ≠ voice) and generates a natural language analysis.
